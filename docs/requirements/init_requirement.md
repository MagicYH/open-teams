# Requirements

我想基于 Agent Client Protocol (ACP) 协议，实现一个开发团队，类似 claude code 的 teams，下面展开说明：

## 基本模块

### 前端

需要有一个 web 端，web 端与后端服务通信

### 后端

使用 python 来开发，python 服务端用于与支持 ACP 协议的客户端通信

### ACP 客户端

使用 ACP 协议与后端服务通信，ACP 客户端可以使用 `opencode acp` 命令来启动

### 持久化

使用 sqlite 来持久化数据

## 功能模块

### 项目管理

1. 需要开发一个 web 页面，用于管理项目
2. 项目可以创建，删除，修改
3. 项目可以设置项目名称
4. 项目可以设置项目目录

### feature 管理

1. 基于项目，可以创建 feature 设置 feature 名称
2. feature 可以创建可以删除

### team 管理

1. 基于项目，可以管理团队成员
2. 可以添加、删除团队成员，团队成员可以设置名称，可以设置角色的 prompt
3. 在启动 ACP 的时候，需要将配置好的 prompt 传入
4. 项目创建的时候需要创建一个默认的 team
5. 默认团队可以从配置文件中读取，使用 yaml 格式
6. 默认团队需要包括 @PM，@Architect，@Developer，@QA
7. 类似的角色可以设置多个，例如 @Developer 可以设置多个，便于后面多个开发人员并行完成开发任务，可以通过不同成员名字来区分，比如 @Developer1，@Developer2

## feature 开发

这个章节会详细描述 feature 开发的需求细节要求

1. 在打开 feature 之后，需要展示一个群聊页面，用户可以在群聊页面中与 team 中的成员进行交流
2. 通过群聊窗口，可以通过 @角色 的方式来向指定角色发送消息
3. 同一条消息中可以 @ 多个角色来向多个角色发送消息
4. 角色在收到消息之后，需要将消息转发给 ACP 客户端，然后开始工作。ACP 客户端需要在项目目录中启动
5. 由于 ACP 客户端是单线程的，但由于并行工作，因此可能会出现 ACP 客户端正在处理其他角色的消息，此时需要将消息加入到队列中，等待 ACP 客户端空闲之后再处理
6. 除了每个 feature 都有一个默认的群聊窗口，每个角色都需要有自己的工作窗口，用于展示角色的工作内容和状态，比如思考过程、工具调用等等，这些数据需要持久化到 db 中
7. 所有角色在完成工作后，都需要通过 @ 某个角色的方式来汇报工作进度，所有有带有 @角色 的消息，都要在群聊窗口中展示
8. 除了团队内定义的角色外，还有 @User 角色，表示用户，所有角色都应该知道还存在其它哪些角色，并且也知道 @User 即用户
9. 与 ACP 客户端通信，需要先创建 session，session 是可以复用的，复用 session 可以使得 ACP 客户单可以在同一个对话上下文内连续工作。
10. session 使用 feature + 角色 来管理，即一个 feature 下一个角色只有唯一的 session，所有其它角色发送给该角色的消息，都要通过排队机制，发送到这个 session 中
11. 因此工作流程是：
    1. 在某个角色收到发给他的消息之后，先基于 feautre_id + role 来从 db 中获取 session
    2. 如果 session 不存在，创建 session，并且持久化到 db 中
    3. 如果 session 存在，那么先测试 session 是否还有效，如果已经失效了，那么按照新建 session 的流程来处理
    4. 如果 session 有效，那么直接基于已有 session 继续工作
12. 角色的群聊消息、自己的工作消息，都需要持久化到 db 中。这样如果用户刷新页面，或者重新打开 feature，消息内容都还可以被正确展示
13. 每个角色在特定 feature 下都需要有一个状态展示栏
    1. 状态栏可以现实现在角色是否正在工作，工作状态现实 busy、空闲状态不做特殊显示
    2. 点击状态栏可以打开这个角色的工作窗口，以便查看角色工作状态
